# 交互式逐段处理工作流

本文档描述 agent（如 DeepSeek TUI）如何使用 `aigc-humanizer-zh` MCP Server
实现 **逐段展示 → 用户决策 → 改写 → 确认替换** 的交互式体验。

## 核心流程

```
┌─────────────┐
│ 用户提交全文  │
└──────┬──────┘
       │
  ┌────▼──────────────────────────────────┐
  │ Step 1: analyze_by_paragraph(text)    │
  │ → 返回每段 AIGC 评分 (0-100) + 命中模式│
  └────┬──────────────────────────────────┘
       │
       │  逐段循环 ──────────────────────┐
       │                                │
  ┌────▼──────────────────────┐         │
  │ Step 2: 展示段 N 的风险     │         │
  │ 「段3 AIGC: 50/100 ⚠」     │         │
  │ 「命中: 段末套路, 模糊归因」 │         │
  │ 「是否改写这一段? [是/否]」  │         │
  └────┬──────────────────────┘         │
       │                                │
       ├── 用户选「否」──────────────────┤
       │  跳过该段，继续下一段            │
       │                                │
       ├── 用户选「是」                   │
       │                                │
  ┌────▼──────────────────────────────┐ │
  │ Step 3: build_rewrite_prompt(     │ │
  │   paragraph, patterns, style)     │ │
  │ → 生成改写 prompt                  │ │
  └────┬──────────────────────────────┘ │
       │                                │
  ┌────▼──────────────────────────────┐ │
  │ Step 4: LLM 执行改写               │ │
  │ (agent 将 prompt 发给 LLM)         │ │
  └────┬──────────────────────────────┘ │
       │                                │
  ┌────▼──────────────────────────────┐ │
  │ Step 5: 展示改写对比                │ │
  │ 「原文: ...」                       │ │
  │ 「改写: ...」                       │ │
  │ 「是否替换为改写版本? [是/否]」     │ │
  └────┬──────────────────────────────┘ │
       │                                │
       ├── 用户选「否」→ 保留原文 ──────┤
       ├── 用户选「是」→ 替换原文 ──────┤
       │                                │
       └── 继续下一段 ──────────────────┘
```

## Agent Prompt 模板

将以下指令作为 system prompt 提供给 agent：

```
你是一个中文学术文本去 AI 味助手，使用 aigc-humanizer-zh MCP 工具进行逐段处理。

对用户提交的文本，严格按以下流程操作：

1. **全文分析**
   调用 analyze_by_paragraph 获取每段 AIGC 评分。

2. **展示概览**
   输出全文摘要：
   - 总段数、高风险段数
   - 整体 AIGC 风险百分比
   - 硬约束命中情况

3. **逐段交互**
   从高风险的段开始，依次处理每一段：

   a) 展示该段信息：
      - 段号、AIGC 评分 (X/100)、风险等级
      - 命中的模式及其具体建议
      - 询问：「这段需要改写吗？」(选项: 是 / 否 / 跳过剩余)

   b) 用户选择「是」：
      - 调用 build_rewrite_prompt 生成改写指令
      - 执行改写
      - 展示原文 vs 改写后的对比
      - 询问：「是否替换为改写版本？」(选项: 是 / 否)

   c) 用户选择「否」或「跳过剩余」：
      - 保留原文，继续下一段（或结束）

4. **最终输出**
   所有段落处理完毕后，输出完整文本。
   可选：调用 assess_quality 展示最终质量评分。
```

## 工具速查

| 步骤 | 工具 | 输入 | 输出 |
|------|------|------|------|
| 全文分析 | `analyze_by_paragraph` | 原始文本 | 每段 AIGC 评分 + 模式命中 |
| 生成改写 prompt | `build_rewrite_prompt` | 段落 + patterns | 结构化改写 prompt |
| 改写后验证 | `assess_quality` | 改写后全文 | 60 分制评分 |
| LaTeX 保护 | `mask_latex` | 含公式文本 | 公式占位文本 |
| LaTeX 还原 | `restore_latex` | 占位文本 | 还原后全文 |

## 实际使用示例

### 用户输入

```
用户: 帮我降低这篇论文的 AI 率

本研究基于框架理论对新媒体环境下的政治传播进行了深入探讨。
依据框架理论，媒体通过选择和强调特定信息，引导受众对政治
议题形成特定认知。

该方案基于三重考量：首先，微博、微信、抖音三平台用户覆盖
面广；其次，三平台内容呈现形式各异，具有对比价值；再次，
相关数据的可及性较好。

研究结果显示，抖音平台的政治内容框架与微博存在显著差异。
该结论印证了框架理论的论断。专家认为，社交媒体在政治动员
中发挥着至关重要的作用。
```

### Agent 交互过程

```
Agent: 已分析全文，共 3 段，整体 AIGC 风险 25%，高风险段 1 个。

───────────────
段 1 | AIGC: 12/100 🟢 | 可跳过
预览: 本研究基于框架理论对新媒体环境下的政治传播进行了深入探讨...
命中: AI高频词「深入探讨」
───────────────
段 2 | AIGC: 12/100 🟢 | 可跳过
预览: 该方案基于三重考量：首先，微博、微信、抖音三平台用户覆盖面广...
命中: 整齐编号逻辑「首先...其次...再次」
───────────────
段 3 | AIGC: 50/100 🔴 | ⚠ 需要改写
预览: 研究结果显示，抖音平台的政治内容框架与微博存在显著差异...
命中:
  • [高] 段末套路结尾: 「该结论印证了」
  • [高] 模糊归因: 「专家认为」
  • [中] 回避系动词: 「发挥着至关重要的作用」

是否改写段 3? [是/否/跳过剩余]
```

用户选择「是」后：

```
Agent: 正在改写段 3...

原文: 研究结果显示，抖音平台的政治内容框架与微博存在显著差异。
      该结论印证了框架理论关于媒介形式影响框架选择的论断，体现
      了新媒体环境对政治传播模式的深刻塑造作用。专家认为，社交
      媒体在政治动员中发挥着至关重要的作用。

改写: 数据显示，抖音和微博的政治内容框架差异明显——抖音的短视频
      叙事逻辑天然依赖情绪驱动，帧选框架的比例远高于微博。这不
      仅仅是「媒介形式影响框架选择」的简单印证，更暴露了平台算
      法对内容形态的深层约束。社交媒体在政治动员中的角色，与其
      说是「发挥重要作用」，不如说是重塑了信息流动的底层规则。

是否替换为改写版本? [是/否]
```

### 含 LaTeX 公式的处理

如果文本含 LaTeX 公式，agent 应先调用 `mask_latex` 保护公式，所有逐段处理在骨架文本上进行，最后调用 `restore_latex` 还原。

```
1. mask_latex(原文) → 骨架文本
2. analyze_by_paragraph(骨架文本) → 逐段分析
3. [逐段改写交互...]
4. restore_latex(改写后骨架) → 最终文本
```

## 阈值调优

- `needs_rewrite` 阈值：默认 AIGC ≥ 25/100 建议改写。可在 `analyze_by_paragraph` 返回的 `needs_rewrite` 字段上调整
- 改写力度：`build_rewrite_prompt` 中的 `style` 参数控制语域
- 质量验收：改写后调用 `assess_quality`，≥ 54 分可提交
